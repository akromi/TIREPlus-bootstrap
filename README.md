## Building the Site

The build system uses a single Node.js script (`build.js`) with zero dependencies — no `npm install` required.

### What it does

1. Reads each page file from `src/pages/` (just the `<main>` content + metadata)
2. Reads the shared partials from `src/_partials/` (head, nav, footer)
3. Assembles them into complete HTML pages
4. Writes the output to `site/` (which gets deployed)

### How to run it

```bash
node build.js
```

That's it. You'll see output like:

```
Building TirePlus site...

  Source:  src/pages
  Output:  site

  ✓  contact-us/index.html
  ✓  index.html
  ✓  tire-search/index.html
  ✓  wheel-search/index.html

Done — 4 pages built.
```

### When to run it

Run `node build.js` after you edit **any file in `src/`**:

| You changed... | Run build? |
|---|---|
| `src/_partials/nav.html` (phone, links) | **Yes** — `node build.js` |
| `src/_partials/footer.html` (hours, links) | **Yes** — `node build.js` |
| `src/_partials/head.html` (fonts, meta) | **Yes** — `node build.js` |
| `src/pages/index.html` (home content) | **Yes** — `node build.js` |
| `src/pages/tire-search/index.html` | **Yes** — `node build.js` |
| `site/css/style.css` (styling) | No |
| `site/js/main.js` (JavaScript) | No |
| `site/assets/js/tireconnect-config.js` | No |
| `site/img/*` (images) | No |

### How page files work

Each page file in `src/pages/` has a front-matter block at the top followed by the `<main>` content:

```html
---
title: Tire Search – Tire Plus | Orleans, ON
description: Search for tires by vehicle or size.
scripts: tireconnect-tires
---
<main id="main-content">
  <!-- only page-specific content goes here -->
</main>
```

Front-matter fields:

| Field | Required | What it does |
|---|---|---|
| `title` | Yes | Sets the `<title>` tag |
| `description` | Yes | Sets the `<meta name="description">` tag |
| `scripts` | No | Injects extra scripts. Values: `tireconnect-tires` or `tireconnect-wheels` |

### GitHub Actions runs the build automatically

The deploy workflow runs `node build.js` before uploading, so you don't need to commit the built `site/*.html` files. However, it's still good practice to run the build locally and commit the results so you can preview before pushing.

## Quick Start (Local Preview)

Build first, then serve:

```bash
# 1. Build pages from src/ into site/
node build.js

# 2. Serve locally (pick one)
cd site && python3 -m http.server 8000       # Python 3
npx serve site                                # Node
# Or: VS Code "Live Server" extension → right-click site/index.html
```

Open `http://localhost:8000` in your browser.

## Setting Up Staging on SiteGround

### 1. Create a Staging Subdomain

1. Log in to **SiteGround → Site Tools → Domain → Subdomains**.
2. Create `staging2.tireplus.ca`.
3. Note the **document root** (e.g. `/home/customer/staging2.tireplus.ca/public_html`).

### 2. Password-Protect Staging

1. **Site Tools → Security → Protected URLs** → add `/` for the staging subdomain.
2. Create a username/password.

*Alternatively*, uncomment the `AuthType` block in `site/.htaccess` and create a `.htpasswd` file:

```bash
htpasswd -c /home/customer/.htpasswd staginguser
```

### 3. Create an FTP Account

1. **Site Tools → FTP Accounts → Create**.
2. Give it access to the staging document root.
3. Note the **FTP hostname**, **username**, and **password** — you'll need them for GitHub secrets.

## GitHub Actions Deployment

### 1. Push the repo to GitHub

```bash
cd tireplus-bootstrap
git init
git add .
git commit -m "Initial commit – Bootstrap 5 rebuild"
git remote add origin git@github.com:YOUR_ORG/tireplus-bootstrap.git
git push -u origin main
```

### 2. Add Repository Secrets

Go to **GitHub → Settings → Secrets and variables → Actions → New repository secret** and add:

| Secret Name    | Value                                                |
|----------------|------------------------------------------------------|
| `FTP_SERVER`   | Your SiteGround FTPS host (e.g. `staging2.tireplus.ca` or the IP) |
| `FTP_USERNAME` | The FTP username you created                         |
| `FTP_PASSWORD` | The FTP password                                     |
| `FTP_REMOTE`   | Remote path, e.g. `/staging2.tireplus.ca/public_html` |

### 3. Deploy

Every push to `main` will auto-deploy. You can also trigger manually:
**GitHub → Actions → Deploy Staging → Run workflow**.

The workflow uses `dangerous-clean-slate: true` so stale files are removed on each deploy.

## Editing Content (Runbook)

### Day-to-day edits

1. **Pull latest code** — `git pull`
2. **Edit** the source file(s) in `src/` (or static files in `site/css`, `site/js`, `site/img`).
3. **Rebuild** (only if you changed files in `src/`) — `node build.js`
4. **Preview locally** (see Quick Start above).
5. **Commit & push** to `main`:

```bash
git add .
git commit -m "Update oil change pricing to $55"
git push
```

6. GitHub Actions runs `node build.js` then deploys automatically. Check the **Actions** tab for status.
7. **Verify** on `https://staging2.tireplus.ca`.

### Common edits

| Task                        | Edit this file                           | Build? |
|-----------------------------|------------------------------------------|--------|
| Change phone number         | `src/_partials/nav.html` + `footer.html` | Yes    |
| Update hours                | `src/_partials/footer.html` + page content if shown in hero/location | Yes |
| Change pricing / specials   | `src/pages/index.html` (highlights bar)  | Yes    |
| Add/remove a service        | `src/pages/index.html` (services grid)   | Yes    |
| Update TireConnect API key  | `site/assets/js/tireconnect-config.js`   | No     |
| Change colors / styling     | `site/css/style.css`                     | No     |
| Update review count         | `src/pages/index.html` (reviews section) | Yes    |

### Tips

- The navbar and footer are defined **once** in `src/_partials/`. No more keeping 4 files in sync.
- Always run `node build.js` after editing anything in `src/`. The GitHub Actions workflow also runs it automatically before deploying.
- Don't edit HTML files in `site/` directly — your changes will be overwritten next time `build.js` runs. Edit the source in `src/` instead.
- Static assets (`site/css/`, `site/js/`, `site/assets/`, `site/img/`) are **not** generated by the build — edit those directly.

## Caching & Cache-Busting

### Staging

The `.htaccess` sets `Cache-Control: no-cache, private` so SiteGround's Dynamic Cache and the browser won't serve stale content on staging.

### If you still see old content on staging

1. **Hard-refresh** the browser: `Ctrl+Shift+R` (Win/Linux) or `Cmd+Shift+R` (Mac).
2. **Purge SiteGround cache**:
   - Site Tools → Speed → Caching → Dynamic Cache → Flush.
   - Or use SG Optimizer if installed.
3. **Purge Cloudflare** (if using): Dashboard → Caching → Purge Everything.

### Production (future)

When going live, update `.htaccess` to enable browser caching for static assets:

```apache
# Production caching
<IfModule mod_expires.c>
  ExpiresActive On
  ExpiresByType text/css             "access plus 1 year"
  ExpiresByType text/javascript      "access plus 1 year"
  ExpiresByType image/png            "access plus 1 year"
  ExpiresByType image/jpeg           "access plus 1 year"
  ExpiresByType image/svg+xml        "access plus 1 year"
  ExpiresByType image/webp           "access plus 1 year"
</IfModule>
```

Use cache-busting query strings: `style.css?v=2` when you change files.

## TireConnect Widget Integration

Both the Tire Search and Wheel Search pages use a shared config + init pattern.

### Shared config (`assets/js/tireconnect-config.js`)

```javascript
window.TC_CONFIG = {
  apiKey: "165d92b73544d5ec4caf11c14e194648",
  locale: "en_CA",
  locationId: 15994,
  locationLock: true
};
```

### Shared init (`assets/js/tireconnect-init.js`)

Reads `window.TC_PAGE.type` to decide whether to call `TCWidget.init()` (tires) or `TCWidget.initWheels()` (wheels). Handles loading/error states automatically.

### Script load order (at bottom of each page)

```html
<script src="/assets/js/tireconnect-config.js"></script>
<script>window.TC_PAGE = { type: "tires" };</script>  <!-- or "wheels" -->
<script src="https://app.tireconnect.ca/js/widget.js"></script>
<script src="/assets/js/tireconnect-init.js"></script>
```

Scripts load **synchronously** (no `defer`) so `TCWidget` exists when the init script runs.

### Changing the API key

Edit **one file**: `assets/js/tireconnect-config.js`. Both pages pick it up automatically.

**Getting your API key:** TireConnect emails your API key after your training session. If you've lost it, contact TireConnect support. The same key works for both tire and wheel widgets.

**Layout shift prevention:** The container `<div>` is wrapped in `.tc-widget-container` which sets `min-height: 600px` so the page doesn't jump when the widget loads.

## Google Reviews Badge

The Home page includes a Google Reviews badge in the "What Our Customers Say" section.

### How it works

A **static fallback** badge is coded in HTML using your Place ID (`ChIJJ3NwUJERzkwRZ97dmGaiNSs`), rating (4.4), and review count (178). This works without any third-party JS.

### To use the RichPlugins dynamic widget

If you want the auto-updating RichPlugins badge (the one from your current WordPress site), add their JS/CSS to the Home page `<head>`:

```html
<!-- RichPlugins Google Reviews -->
<link href="https://cdn.richplugins.com/css/flavor.min.css" rel="stylesheet">
<script src="https://cdn.richplugins.com/js/flavor.min.js" defer></script>
```

Then replace the static badge `<div>` with the RichPlugins embed code from your dashboard.

### Updating review count manually

If using the static badge, periodically update the `data-count` attribute and the "178" text in `index.html`.

## Contact Form Setup

The contact form at `contact-us/index.html` is a static HTML form. Choose one approach:

### Option A – Formspree (recommended, free tier available)

1. Sign up at [formspree.io](https://formspree.io).
2. Create a form → get your form ID (e.g. `xvgpknpw`).
3. Update the `<form>` tag:

```html
<form class="contact-form" action="https://formspree.io/f/xvgpknpw" method="POST">
```

4. Done — submissions go to your email.

### Option B – SiteGround PHP handler

Create `site/contact-handler.php`:

```php
<?php
$to      = 'info@tireplus.ca';
$subject = 'Website Contact: ' . htmlspecialchars($_POST['name'] ?? 'Unknown');
$body    = "Name: {$_POST['name']}\nEmail: {$_POST['email']}\nPhone: {$_POST['phone']}\nVehicle: {$_POST['vehicle']}\n\nMessage:\n{$_POST['message']}";
$headers = "From: noreply@tireplus.ca\r\nReply-To: {$_POST['email']}";
mail($to, $subject, $body, $headers);
header('Location: /contact-us/?sent=1');
?>
```

Update the form action to `/contact-handler.php`.

## Going Live (Production Cut-Over)

When staging is approved:

1. **Back up** the live WordPress site (files + database) via SiteGround.
2. **Update `.htaccess`** — switch from no-cache to production caching rules (see Caching section above).
3. **Update GitHub secrets** — change `FTP_REMOTE` to point to the production document root.
4. **Push to main** — the workflow deploys to production.
5. **Test** all pages, TireConnect widgets, contact form, map, phone links.
6. **DNS** — no changes needed if the domain already points to SiteGround.
7. **SSL** — SiteGround auto-provisions Let's Encrypt. Verify HTTPS works.

> **Do not** change production DNS or the live WordPress site until you are explicitly ready.

## Accessibility Checklist

The templates include these accessibility features:

- [x] **Skip link** (`<a class="skip-link">`) on every page
- [x] **Semantic headings** (h1 → h6, one h1 per page)
- [x] **Landmark roles** (`<nav>`, `<main>`, `<footer>`)
- [x] **ARIA labels** on navigation, sections, and icon-only links
- [x] **Visible focus** (`:focus-visible` outline on all interactive elements)
- [x] **Keyboard-navigable** (offcanvas menu, links, buttons)
- [x] **Alt text** on images
- [x] **Color contrast** (navy/white, red/white meet WCAG AA)
- [x] **Responsive** (mobile-first Bootstrap grid)
- [x] **`aria-current="page"`** set dynamically on active nav link

### Testing

- Run [axe DevTools](https://www.deque.com/axe/) in Chrome for automated checks.
- Tab through every page to verify keyboard access.
- Test with a screen reader (NVDA on Windows, VoiceOver on Mac).
